# P1.5：认证与授权设计基线

- 状态：**Current / Accepted**
- Slice：`P1.5 S00`
- 权威仓库：`mom-platform`
- 适用仓库：`mom-platform`、`mom-web`、`mom-mobile`
- 生效日期：2026-07-19

> 本文是 MOM P1.5 认证与授权闭环的跨仓库权威协议。浏览器和移动端仓库只定义各自运行时实现细节，不得重新定义后端安全语义。

## 1. 阶段定位

Phase 01 已完成基础技术骨架、可观测性、数据访问、消息与中间件兼容性验证，但尚未实现完整 IAM、Authorization Server、Gateway Resource Server、业务服务 Resource Server、用户授权 Session、Refresh Rotation、RBAC 管理、Web 登录或 Mobile 登录闭环。

P1.5 的目标是补齐认证、授权、数据范围、Session 撤销、管理 API、Web/Mobile Runtime 和安全 E2E。S00 只冻结设计与实施计划，不实现正式业务代码或 Flyway DDL。

## 2. 权威边界

| 领域 | 权威来源 |
|---|---|
| 用户、账号、OAuth/OIDC、Token、Session、RBAC、Factory/Party Scope、安全审计、Gateway/Resource Server 契约 | `mom-platform` |
| Web PKCE、Token 内存管理、前端访问控制、错误恢复、当前 Factory 偏好 | `mom-web` |
| Mobile 系统浏览器登录、App Link、安全存储、冷启动恢复、离线命令归属 | `mom-mobile` |

## 3. 用户与主体模型

统一使用 `iam_user` 账号模型。

`user_type` 仅允许：

- `INTERNAL`
- `SUPPLIER`
- `CUSTOMER`

规则：

1. PDA 是客户端，不是用户类型。
2. `user_type` 与 Role 严格分离。
3. 一个 `SUPPLIER` 用户只能绑定一个 `supplier_id`。
4. 一个 `CUSTOMER` 用户只能绑定一个 `customer_id`。
5. 一个账号不能同时绑定多个供应商或客户主体。
6. 一个供应商或客户主体可以拥有多个用户账号。
7. `INTERNAL` 用户不得登录 Supplier Portal 或 Customer Portal。
8. 内部人员代办外部业务只能通过 MOM Admin，不得切换或冒充外部身份。
9. 内部代办必须记录 `actor_user_id`、`represented_party_type`、`represented_party_id`、`delegation_reason`，并使用独立 Permission。

## 4. OAuth Client 与应用访问矩阵

预置四个 Public Client：

| Client ID | 应用 | 允许用户 |
|---|---|---|
| `mom-admin-web` | MOM Admin | `INTERNAL` |
| `mom-supplier-web` | Supplier Portal | `SUPPLIER` |
| `mom-customer-web` | Customer Portal | `CUSTOMER` |
| `mom-mobile-pda` | MOM Mobile PDA | 仅具备独立 Mobile Access 授权的 `INTERNAL` |

Client ID 表示应用，不表示用户。

统一采用：

- Authorization Code
- PKCE `S256`
- OpenID Connect
- Public Client
- 无 `client_secret`
- Redirect URI 精确匹配
- 不开放动态 Client 注册

所有用户仍通过账号密码登录，但密码只提交给 IAM 登录页面。

## 5. Cookie 与 API 凭证

1. P1.5 不建设 BFF。
2. 浏览器和移动端调用 Gateway API 时统一使用 Bearer Access Token。
3. 不使用应用 Session Cookie 调用 Gateway API。
4. IAM 可以用自己的 HttpOnly Cookie 维护 IAM 登录页面状态和基础 SSO。
5. IAM Cookie 不等于应用 Session，也不是 Gateway API 凭证。
6. IAM 登录表单自身必须提供 CSRF 防护。

## 6. Token 与 Claims

### 6.1 Token 类型

- Access Token：短期 JWT，初始有效期 10 分钟。
- Refresh Token：高熵 Opaque Token。
- ID Token：只用于 OIDC 客户端身份信息，不得调用业务 API。

### 6.2 用户授权 Session

- Web Session 绝对有效期：8 小时。
- Mobile Session 绝对有效期：12 小时。
- Refresh Rotation 不延长绝对 Session 有效期。

### 6.3 Refresh Token 安全

1. Refresh Token 明文不得写入数据库、日志、Trace 或安全审计。
2. 数据库存储 Refresh Token 的 HMAC-SHA-256 摘要。
3. 一个 Session 最多存在一个 `ACTIVE` Refresh Token。
4. Refresh 使用数据库事务和行锁。
5. 不实现 Refresh Token 宽限窗口。
6. 旧 Refresh Token 重放会把整个 Session 标记为 `COMPROMISED`，撤销当前 ACTIVE Token，并将 `sid` 写入 Redis。

### 6.4 JWT Claims

Access Token 至少包含：

- `iss`
- `sub`
- `aud`
- `iat`
- `nbf`
- `exp`
- `jti`
- `sid`
- `client_id`
- `user_type`
- `roles`
- `permissions`
- `factory_ids`
- `party_type`
- `party_id`

OAuth Scope 与业务 Permission 必须分离，不把大量业务 Permission 注册成 OAuth Scope。

## 7. RBAC 与 Permission

采用标准模型：`User → Role → Permission`。

规则：

1. 用户不能直接分配 Permission。
2. 不支持角色继承、Deny Permission 或 ABAC。
3. 一个用户可以拥有多个角色；有效权限是全部有效角色权限的并集。
4. Role 必须限定 `applicable_user_type`，禁止跨 `INTERNAL`、`SUPPLIER`、`CUSTOMER` 分配。
5. Role 表示职责，Permission 表示具体业务能力。
6. Permission Code 格式为 `domain:resource:action`。
7. Role 与 Factory Scope 分离。
8. Permission 由代码和 Flyway 预置。
9. 管理端只能查看 Permission 目录，不能动态创建或修改 Permission Code。
10. 系统角色不可删除；管理员可创建自定义角色并配置系统 Permission。

## 8. Factory Scope 与 Party Scope

P1.5 的通用数据权限只实现 Factory Scope 和 Party Scope。

1. 一个用户可以拥有多个 Factory Scope。
2. 外部用户只能绑定一个供应商或客户主体。
3. 外部用户的 Factory Scope 必须是该主体有效业务关系工厂的子集。
4. Factory 主数据属于 MDM；IAM 只保存 `factory_id`。
5. JWT 中保存 `factory_ids`。
6. 当前工厂使用 `X-Factory-Id` 传递。
7. `X-Factory-Id` 只是当前工作上下文，不是授权证明。
8. 创建业务对象时，`factory_id` 必须由服务端上下文确定。
9. 外部对象的 `supplier_id` 或 `customer_id` 必须来自 JWT `party_id`，前端不得自由切换。
10. 内部代办是唯一例外，但必须有独立 Permission 和明确审计。
11. Factory 和 Party 条件应尽量直接进入 SQL 查询条件。
12. 对象属于其他 Factory、Supplier 或 Customer 时，对外返回 404，避免对象枚举。

P1.5 不建设 Warehouse、Workshop、Line、Department 等通用 Scope，不建设 SQL 自动改写数据权限 DSL，也不支持同一用户在不同 Factory 使用不同 Role。

## 9. 安全职责边界

### 9.1 IAM

IAM 负责：

- 用户认证与 Client 校验
- 用户、角色、权限、Factory Scope、Party Binding
- Authorization Server、Token、Session、Refresh Rotation 与撤销
- 安全审计
- `/api/iam/me`
- IAM 管理 API

### 9.2 Gateway

Gateway 负责：

- JWT 签名、Issuer、Audience、有效期校验
- 公开与受保护路由
- Client 粗粒度入口隔离
- revoked `sid` 检查
- CORS、路由、限流、Correlation ID、Trace
- 删除客户端伪造的 `X-MOM-*` Header
- 原样转发 Bearer JWT

Gateway 不负责最终业务 Permission、Factory/Party 对象归属、业务状态机或动态菜单。

Redis 不可用时，受保护 API 必须 Fail Closed。

### 9.3 业务服务

每个业务服务作为 Resource Server，再次验证 Token，并负责：

- 最终业务 Permission
- Factory Scope
- Party Scope
- 业务对象归属
- 领域状态与最终授权
- 业务操作审计

## 10. Session 与撤销

1. JWT `sid` 对应 `iam_user_session`。
2. 一个用户授权 Session 对应一条 Refresh Token 轮换链，不单独创建 Refresh Token Family 表。
3. 以下事件必须撤销相关 Session：用户禁用、密码修改、密码重置、主体绑定失效、Client 禁用、Mobile Access 关闭、设备丢失、Refresh Token 重放。
4. 普通 Role、Permission、Factory Scope 变更在下一次 Token 刷新后生效。
5. 高风险变更提供“保存并强制重新登录”。

## 11. Web Runtime 契约

`mom-web` 必须形成：

- `@mom/auth`
- `@mom/access`
- `@mom/api-client`

规则：

1. Access Token 和 Refresh Token 只存在内存。
2. `sessionStorage` 只保存 `state`、`nonce`、`code_verifier`、`returnUrl`。
3. Callback 后立即清理一次性事务，Code 交换后从地址栏移除 Authorization Code。
4. 页面刷新后重新执行顶层 PKCE，利用 IAM 登录状态快速恢复。
5. 不使用 localStorage、IndexedDB、持久化 Pinia、普通 Cookie 保存应用 Token。
6. 不使用 hidden iframe silent renew 或跨标签页 Token 同步。
7. 必须实现 Single Flight Refresh；每个业务请求最多自动重试一次。
8. 403 不触发 Token Refresh。
9. `/api/iam/me` 是正式前端权限上下文来源。
10. 当前 Factory 可保存为普通偏好，但启动时必须重新校验。
11. Supplier Portal 和 Customer Portal 不提供主体切换器。

## 12. Mobile Runtime 契约

1. 使用系统浏览器登录，不在嵌入式 WebView 输入账号密码。
2. 使用 Authorization Code + PKCE，正式环境优先 HTTPS App Link 回调。
3. Access Token 保存在内存；Refresh Token 使用 Android 安全存储。
4. PKCE 临时事务必须承受 App 进程被回收。
5. 冷启动优先用 Refresh Token 恢复 Session。
6. Refresh Token 本地替换结果不确定时，不得重复使用旧 Token。
7. H5 只用于开发调试，按 Web 模型处理，不持久化真实 Refresh Token。
8. 离线命令必须记录创建用户、Session、Client、Factory、Party、所需 Permission 和创建时间。
9. 用户 A 的离线命令不能由用户 B 自动同步。

## 13. IAM 管理端与安全审计

P1.5 最小管理能力：

- 用户管理
- 用户角色分配
- Factory Scope
- Mobile Access
- 角色管理
- Permission 只读目录
- Session 管理
- 安全审计
- OAuth Client 只读查看，以及可选受控启用/禁用

用户状态只使用 `ENABLED`、`DISABLED`；账号锁定使用 `failed_login_count`、`locked_until`。

用户删除采用逻辑删除并撤销全部 Session。管理员密码重置后，首次登录必须修改密码并撤销全部 Session。

外部主体重新绑定必须使用独立权限、填写原因、撤销全部 Session，并写完整安全审计。

安全审计与业务审计分离，且不得保存密码、密码摘要、Access Token、Refresh Token、Refresh Token 摘要、Authorization Code、`code_verifier`、完整 Authorization Header 或 JWT 私钥。

建议基础管理角色：`PLATFORM_ADMIN`、`IAM_ADMIN`、`SECURITY_AUDITOR`。系统必须始终保留至少一个有效 `PLATFORM_ADMIN`。

## 14. CurrentActor 与 MyBatis-Plus 审计

`mom-core` 定义：

- `AuditActor`
- `ActorType`
- `CurrentActorProvider`
- `AuditContextExecutor`

`mom-security` 从 JWT/SecurityContext 提供当前用户 Actor；`mom-data` 实现 MyBatis-Plus `MetaObjectHandler`，且不得依赖 `mom-security`。

普通 INSERT 自动填充 `created_at`、`created_by`、`updated_at`、`updated_by`；普通 UPDATE 只填充 `updated_at`、`updated_by`。

统一使用 UTC `Instant` 和可注入 `Clock`。找不到用户 Actor 时不能静默使用 SYSTEM；后台任务必须显式建立 SYSTEM Actor。

`factory_id`、`party_id`、`supplier_id`、`customer_id` 等业务归属字段不得由 `MetaObjectHandler` 自动填充。必须禁止或显式补齐可能绕过自动填充的 `update(wrapper)` 等路径。

乐观锁使用 `@Version` 与 `OptimisticLockerInnerInterceptor`。追加型安全审计表、OAuth 协议表和特殊状态表不强制继承完整 BaseEntity。

## 15. 数据库 DDL 强制规范

S00 不创建正式 DDL。P1.5 后续所有 PostgreSQL Flyway DDL 必须包含完整、准确的中文注释，至少覆盖：

- 表中文注释和每个字段中文注释
- 枚举/状态字段全部取值说明
- 唯一约束的业务目的
- 关键索引的查询或并发用途
- JSON/JSONB 结构与使用边界
- UTC 时间说明
- 金额、数量、单位字段业务语义
- Token 摘要等安全字段的安全边界
- 数据保留或清理字段语义

禁止只写“编码”“状态”“时间”“版本”“备注”等无效注释。缺少中文注释视为 Slice 未完成。

## 16. 跨仓库协议

前端和移动端必须以本文件为后端协议权威来源。任何 Client ID、user_type、Token 生命周期、Claims、`X-Factory-Id`、Party Binding、Session 或职责边界变更，必须先修改 `mom-platform` ADR 与本基线，再同步 `mom-web`、`mom-mobile`。

## 17. S00 完成判定

S00 完成必须满足：

- 三仓库完成真实状态盘点和文档修改
- 三仓库各自创建独立 PR
- Client ID、user_type、Token、Session、职责边界和 Slice 编号一致
- 不把前端判断或 `X-Factory-Id` 描述为授权证明
- 不保留仍标记为 Current/Active 的 BFF 主方案
- 不实现 Java、Vue、TypeScript、正式 DDL 或 P1.5 后续 Slice 代码
