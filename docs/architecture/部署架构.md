# 部署架构

## 1. 目标环境

V1 部署到三节点 k3s：

```text
k3s-control-plane
k3s-worker-01
k3s-worker-02
```

开发环境可以使用本地进程或 Docker Compose 启动中间件，但最终验收以 k3s 为准。

## 2. 仓库职责

- `mom-platform`：应用代码、Dockerfile 约定、应用级 Helm Values 示例。
- `mom-infra`：k3s Namespace、中间件、Ingress、可观测性、部署模板和运维脚本。
- 业务仓库不得复制完整基础设施配置，使用版本化部署契约协作。

## 3. Namespace 建议

| Namespace | 内容 |
|---|---|
| `mom-system` | Gateway、IAM、MDM、Integration |
| `mom-business` | MES、WMS、QMS、EMS、EAM、Traceability |
| `mom-device` | PCS、WCS 和模拟器 |
| `mom-data` | PostgreSQL、Redis、RocketMQ、Nacos、Seata |
| `mom-observability` | OTel Collector、Tempo、Loki、Prometheus、Grafana |

V1 可根据资源限制合并 Namespace，但名称和网络边界应保持清晰。

## 4. 应用部署规范

每个服务至少定义：

- Deployment。
- Service。
- ConfigMap。
- Secret 引用。
- Startup Probe。
- Liveness Probe。
- Readiness Probe。
- CPU/Memory Request 与 Limit。
- PodDisruptionBudget（核心服务）。
- 滚动升级策略。

## 5. 副本建议

| 服务 | V1 副本 |
|---|---:|
| Gateway | 3 |
| IAM | 2 |
| MDM | 2 |
| MES | 2 |
| WMS | 2 |
| QMS | 2 |
| Integration | 2 |
| Traceability | 2 |
| EMS/EAM | 1 |

该配置用于演示多实例、限流和滚动升级，不代表生产容量规划。

## 6. 配置和密钥

- 普通配置通过 `application.yml` 默认值、Nacos Config 和 ConfigMap 分层管理。
- 密钥使用 Secret，不进入 Git。
- 配置必须区分 local、test、k3s 和后续生产环境。
- Nacos Config 使用 `spring.config.import`，不使用旧 Bootstrap 配置方式。
- 镜像内不得固化环境地址和凭证。

## 7. 网络入口

```text
用户/外部系统
→ Ingress Controller
→ mom-gateway Service
→ 内部服务
```

- 领域服务默认不直接暴露到集群外。
- Grafana 和管理组件使用受控管理入口。
- PostgreSQL、Redis、RocketMQ 和 Nacos 不对公网开放。

## 8. 存储

- PostgreSQL、RocketMQ、Nacos 和可观测性后端使用持久卷。
- Redis V1 根据用途决定持久化，但不得把 Redis 当作唯一权威数据源。
- PVC 和备份策略在 `mom-infra` 中定义。
- 数据恢复演练至少覆盖 PostgreSQL 备份恢复和应用重新部署。

## 9. 发布策略

### 滚动升级

- Readiness Probe 未通过的 Pod 不接收流量。
- 新旧版本需要保持 API、事件和数据库向前兼容。
- Flyway 采用扩展—迁移—收缩策略。

### 回滚

- 应用镜像使用不可变版本标签。
- Helm/Manifest 保留历史版本。
- 数据库破坏性变更不得与应用版本同时直接执行。
- 回滚后需要验证消息消费、Outbox 和业务状态。

## 10. 故障演练

V1 必须演练：

- 删除一个 Gateway Pod，流量继续服务。
- 删除一个 MES/WMS Pod，消息和请求恢复。
- Redis 短时中断，观察限流降级策略。
- RocketMQ 重复投递和消费者重启。
- Nacos 短时不可用。
- PCS/WCS 模拟器断线。
- 应用滚动升级和版本回滚。
- 一个 Worker 节点不可用后的 Pod 迁移。

## 11. 资源约束

本地三节点集群资源有限，V1 应：

- 对所有服务设置合理 Request/Limit。
- EMS/EAM 使用单副本和最小功能。
- 降低 Tempo/Loki 数据保留周期。
- 避免在 V1 引入 Kafka、Flink、Elasticsearch 和 IoTDB。
- 使用压测窗口而非长期高负载运行。
