# 数据架构

## 1. 总体原则

- PostgreSQL 17 是 V1 业务权威数据存储。
- 每个服务使用独立 Schema。
- 禁止跨 Schema JOIN、跨域写入和共享 Mapper。
- Redis 只保存可重建状态，不作为库存、检验、工单等最终权威来源。
- 所有数据库结构变更由 Flyway 管理。

## 2. Schema 规划

| 服务 | Schema 建议 |
|---|---|
| IAM | `iam` |
| MDM | `mdm` |
| MES | `mes` |
| WMS | `wms` |
| QMS | `qms` |
| EMS | `ems` |
| EAM | `eam` |
| Integration | `integration` |
| Traceability | `traceability` |

服务运行账号原则上只具有自身 Schema 的读写权限。

## 3. 时间与标识

- 技术时间戳统一使用 UTC 存储。
- 领域中的时间点优先使用 `Instant`。
- 业务日期使用 `LocalDate` 并同时保存工厂时区或业务时区配置。
- 查询时间范围转换为 UTC 半开区间 `[start, end)`。
- 业务主键采用统一的字符串或长整型策略，选型由后续 ID ADR 固化。
- 外部业务单号和内部主键必须分离。

## 4. 库存模型

### 4.1 事实流水

库存每次变化形成不可变流水，例如：

- 收货。
- 上架。
- 移库。
- 预占。
- 释放预占。
- 锁定和解锁。
- 生产领用和退料。
- 成品入库。
- 拣货和发运。
- 盘点调整。

### 4.2 实时余额

余额至少区分：

```text
on_hand      现有量
reserved     预占量
locked       锁定量
quarantine   隔离量
available    可用量
```

基础关系：

```text
available = on_hand - reserved - locked - quarantine
```

余额是为查询性能维护的派生状态，必须能够由事实流水重算和对账。

### 4.3 并发控制

- 写入使用数据库事务。
- 余额更新使用版本号、条件更新或明确的行锁策略。
- 接口使用业务幂等键。
- 不允许仅依赖 Redis 锁保证库存最终正确。

## 5. 批次谱系模型

最小对象：

- 工单。
- 原料批次。
- 半成品批次。
- 成品批次。
- 容器。
- 投料消耗。
- 生产产出。
- 谱系边。

谱系边至少包含：

```text
id
input_batch_id
output_batch_id
operation_type
quantity
unit
production_order_id
process_step_id
container_id
occurred_at
event_id
```

谱系边原则上不可覆盖修改；纠错通过冲销、更正或新事实表达。

## 6. 生产版本快照

工单下达时固化：

- 配方版本。
- BOM 版本。
- 工艺版本。
- 质量标准版本。
- 关键主数据摘要。

快照创建后不受 MDM 后续修改影响。

## 7. Outbox / Inbox

每个需要可靠发布事件的服务维护自身 Outbox；每个需要消费去重的服务维护 Inbox。

Outbox 至少记录：

- `event_id`。
- 事件类型和版本。
- 聚合类型和聚合 ID。
- Payload。
- 创建时间、发布时间、重试次数和状态。

Inbox 至少记录：

- `event_id`。
- 消费者名称。
- 首次处理时间。
- 处理结果和错误摘要。

## 8. 查询模型

跨领域复杂查询使用以下方式之一：

1. 调用多个 Query API 后由应用层组合。
2. 通过事件维护本地只读查询模型。
3. 由 Traceability 等明确的查询服务维护跨域投影。

禁止通过跨 Schema JOIN 快速绕过边界。

## 9. 数据迁移规范

- Flyway 脚本按服务和版本管理。
- 已发布迁移禁止直接修改。
- 滚动发布期间优先使用“扩展—迁移—收缩”方式。
- 删除字段和表必须经过至少一个兼容版本。
- 关键迁移需要备份、回滚或补偿方案。

## 10. 数据保留与脱敏

- 审计、库存流水和谱系事实按长期保留设计。
- Trace 和日志按环境设置保留周期。
- Token、密码、密钥和敏感个人信息不得进入事件 Payload、日志或 Trace。
- 测试数据使用虚构企业、人员和批次。
