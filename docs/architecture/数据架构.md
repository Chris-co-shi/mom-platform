# 数据架构

## 1. 总体原则

- PostgreSQL 17 是 V1 业务权威数据存储。
- 每个服务使用独立 Schema。
- 禁止跨 Schema JOIN、跨域写入和共享 Mapper。
- Redis 只保存可重建状态，不作为库存、检验、工单等最终权威来源。
- 所有数据库结构变更由 Flyway 管理。

## 2. Schema 规划

| 服务 | Schema 建议 |
|---|---|
| IAM | `iam` |
| MDM | `mdm` |
| MES | `mes` |
| WMS | `wms` |
| QMS | `qms` |
| EMS | `ems` |
| EAM | `eam` |
| Integration | `integration` |
| Traceability | `traceability` |

服务运行账号原则上只具有自身 Schema 的读写权限。

### 2.1 DataSource 与连接池

- 一个领域服务默认只创建一个权威 `DataSource`，并只连接自身 Schema。
- 一个应用实例默认只创建一个 HikariCP 连接池，不引入动态数据源 Starter 或隐式路由。
- 业务表、Outbox 表和 Inbox 表共用同一 DataSource、事务管理器和本地事务。
- Outbox Publisher 与所属服务共用连接池，不单独创建“消息数据库连接池”。
- 外部遗留库、报表库和读副本是例外场景，必须通过独立 ADR 明确只读、一致性、事务、健康检查和故障策略。
- 默认 HikariCP 参数为：`minimumIdle=1`、`maximumPoolSize=5`、`connectionTimeout=3000ms`、`validationTimeout=2000ms`。
- PgJDBC 启用 TCP Keepalive，并使用稳定的 `ApplicationName` 供 PostgreSQL `pg_stat_activity` 识别服务连接。
- 数据库会话通过连接初始化 SQL 固定为 UTC。
- `maxLifetime` 和 `keepaliveTime` 在数据库代理、网络设备和连接寿命上限明确前不覆盖 Hikari 默认值。
- 泄漏检测默认关闭，仅在诊断环境临时启用。

连接池容量必须结合最大 Pod 副本数评估：

```text
应用最大连接数
= Σ（服务最大副本数 × 每实例 maximumPoolSize）
+ Flyway、任务、监控和运维连接
+ 滚动升级与故障扩容预留
```

应用不得占满 PostgreSQL 全部连接槽位，必须为运维、迁移、监控和滚动发布保留容量。

## 3. 时间与标识

- 技术时间戳统一使用 UTC 存储。
- 领域中的时间点优先使用 `Instant`。
- 业务日期使用 `LocalDate` 并同时保存工厂时区或业务时区配置。
- 查询时间范围转换为 UTC 半开区间 `[start, end)`。
- Java 技术主键统一使用 `String`，PostgreSQL 使用 `varchar(19)`，JSON 中不得输出为 Number。
- 外部业务单号、物料编码、工单号和内部技术主键必须分离。

## 4. 库存模型

### 4.1 事实流水

库存每次变化形成不可变流水，例如：

- 收货。
- 上架。
- 移库。
- 预占。
- 释放预占。
- 锁定和解锁。
- 生产领用和退料。
- 成品入库。
- 拣货和发运。
- 盘点调整。

### 4.2 实时余额

余额至少区分：

```text
on_hand      现有量
reserved     预占量
locked       锁定量
quarantine   隔离量
available    可用量
```

基础关系：

```text
available = on_hand - reserved - locked - quarantine
```

余额是为查询性能维护的派生状态，必须能够由事实流水重算和对账。

### 4.3 并发控制

- 写入使用数据库事务。
- 余额更新使用版本号、条件更新或明确的行锁策略。
- 接口使用业务幂等键。
- 不允许仅依赖 Redis 锁保证库存最终正确。

## 5. 批次谱系模型

最小对象：

- 工单。
- 原料批次。
- 半成品批次。
- 成品批次。
- 容器。
- 投料消耗。
- 生产产出。
- 谱系边。

谱系边至少包含：

```text
id
input_batch_id
output_batch_id
operation_type
quantity
unit
production_order_id
process_step_id
container_id
occurred_at
event_id
```

谱系边原则上不可覆盖修改；纠错通过冲销、更正或新事实表达。

## 6. 生产版本快照

工单下达时固化：

- 配方版本。
- BOM 版本。
- 工艺版本。
- 质量标准版本。
- 关键主数据摘要。

快照创建后不受 MDM 后续修改影响。

## 7. Outbox / Inbox

每个需要可靠发布事件的服务维护自身 Outbox；每个需要消费去重的服务维护 Inbox。

业务写入与 Outbox 写入必须位于同一个 DataSource 和同一个本地事务中。消费端业务写入与 Inbox 去重记录同样必须位于同一个 DataSource 和本地事务中。不得为 Outbox 或 Inbox 创建独立连接池，从而破坏本地事务原子性。

Outbox 至少记录：

- `event_id`。
- 事件类型和版本。
- 聚合类型和聚合 ID。
- Payload。
- 创建时间、发布时间、重试次数和状态。

Inbox 至少记录：

- `event_id`。
- 消费者名称。
- 首次处理时间。
- 处理结果和错误摘要。

Outbox Publisher 领取记录和状态变更必须使用短事务；调用 RocketMQ 的网络过程不得持续持有数据库行锁和连接。

## 8. 查询模型

跨领域复杂查询使用以下方式之一：

1. 调用多个 Query API 后由应用层组合。
2. 通过事件维护本地只读查询模型。
3. 由 Traceability 等明确的查询服务维护跨域投影。

禁止通过跨 Schema JOIN 快速绕过边界。

## 9. 数据迁移规范

- Flyway 脚本按服务和版本管理。
- 已发布迁移禁止直接修改。
- 滚动发布期间优先使用“扩展—迁移—收缩”方式。
- 删除字段和表必须经过至少一个兼容版本。
- 关键迁移需要备份、回滚或补偿方案。

## 10. 数据保留与脱敏

- 审计、库存流水和谱系事实按长期保留设计。
- Trace 和日志按环境设置保留周期。
- Token、密码、密钥和敏感个人信息不得进入事件 Payload、日志或 Trace。
- 测试数据使用虚构企业、人员和批次。
