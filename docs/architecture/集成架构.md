# 集成架构

## 1. 集成原则

- 外部系统只能通过 Gateway 和 Integration Hub 访问 MOM。
- 领域服务之间的即时查询和校验使用同步调用。
- 状态传播、长流程和跨系统协作优先使用领域事件。
- 所有消息消费者按“至少一次投递”设计，必须实现幂等。
- 不使用长时间 Seata 全局事务包裹完整制造流程。
- Spring Cloud Stream 负责消息 Binding 和 Broker 适配，不承担数据库本地事务与业务幂等。

## 2. 技术选择矩阵

| 场景 | 方式 | 说明 |
|---|---|---|
| 单次查询、即时校验 | OpenFeign | 设置超时、错误映射和 Trace 传播 |
| 状态变化通知 | Spring Cloud Stream + RocketMQ | Outbox 可靠发布，Inbox 去重，Binder 负责 Broker 适配 |
| 外部同步接口 | Gateway + Integration Hub | 认证、映射、审计、限流和幂等 |
| 设备命令和结果 | RocketMQ/API | 使用 `command_id` 和状态机 |
| 模拟遥测 | MQTT | 非关键控制数据，不替代领域事件 |
| 单服务数据库事务 | 本地事务 | 默认选择 |
| 短跨库事务 PoC | Seata AT | 严格限制范围和持续时间 |
| 关键资源冻结 | TCC | 只在预占等明确资源场景评估 |
| 长业务流程 | Saga 思想 + 事件 + 补偿 | 每一步有明确状态、重试和人工处理 |

## 3. 统一事件信封

所有跨服务事件至少包含：

```json
{
  "eventId": "全局唯一事件标识",
  "eventType": "inventory.received",
  "eventVersion": 1,
  "occurredAt": "UTC 时间",
  "producer": "mom-wms-server",
  "aggregateType": "InventoryBatch",
  "aggregateId": "业务聚合标识",
  "correlationId": "业务关联标识",
  "payloadJson": "版本化 JSON 文本"
}
```

约束：

- 事件名称使用已经发生的业务事实。
- Payload 不得直接复用数据库 Entity。
- 事件版本升级必须保持消费者兼容或提供新 Topic/Type。
- 敏感信息不得进入事件、Header、日志或错误摘要。
- `eventId` 在首次写入 Outbox 时生成，重试不得重新生成。
- Java 与数据库中的事件 ID 使用字符串，跨语言消费者不得按数字解析。
- 当前基础类型使用 `payloadJson` 保存明确 JSON 文本；后续 Schema Registry 由独立 ADR 评估。

## 4. Spring Cloud Stream 与 RocketMQ 边界

```text
领域应用 / Outbox Publisher
        ↓ EventTransport
Spring Cloud Stream StreamBridge / Function
        ↓ Binding
Spring Cloud Alibaba RocketMQ Binder
        ↓
RocketMQ NameServer / Broker
```

职责：

- `mom-messaging` 定义 Broker 无关的事件信封、Header 和 `EventTransport`。
- `StreamBridge` 使用稳定 Binding 名称发送，不根据 HTTP 参数或消息内容无限创建动态 Binding。
- 函数式 `Consumer<Message<EventEnvelope>>` 负责接收和调用应用服务。
- Topic、Producer Group、Consumer Group、NameServer、超时、重新消费和 DLQ 通过服务配置显式声明。
- Producer 使用同步发送结果；StreamBridge 返回 `true` 只表示传输被接受，不表示消费者业务完成。
- Spring Cloud Stream 通用消费者 `maxAttempts` 默认设为 1，避免与 RocketMQ Broker 重新消费叠加。
- RocketMQ 持续消费失败最终进入 `%DLQ%<consumer-group>`；DLQ 不等于业务补偿完成。

禁止：

- 领域 API 或 Entity 引用 RocketMQ Client、Binder 或 `StreamBridge` 类型。
- 使用 Binder 事务替代数据库 Outbox。
- 在持有数据库行锁的事务中等待 Broker 网络响应。
- 把匿名消费者组、自动生成 Topic 或无限动态 Binding 用于正式业务。

## 5. Outbox 发布流程

```text
业务写入 + Outbox INSERT（同一 DataSource、本地事务）
→ 提交成功，状态 PENDING
→ 短事务 FOR UPDATE SKIP LOCKED 领取
→ 写入 CLAIMED + lease_owner + lease_until
→ 提交事务，释放连接和行锁
→ 事务外通过 StreamBridge 同步发送 RocketMQ
→ 新短事务 CAS 标记 SENT
```

发送失败：

```text
CLAIMED
→ CAS 更新 RETRY + retry_count + next_attempt_at
→ 指数退避后重新领取
→ 超过最大次数进入 DEAD
```

必须处理：

- 应用在业务提交后、消息发送前崩溃。
- 发布实例领取后崩溃，由租约过期恢复。
- 消息发送成功但 Outbox 状态更新失败。
- 重复发送同一 `eventId`。
- Broker 中断和恢复。
- 长时间 PENDING/RETRY、租约超时和 DEAD 告警。

Outbox Publisher 与所属领域服务共用唯一 HikariCP 连接池，不创建第二 DataSource。

## 6. Inbox 消费流程

```text
RocketMQ 至少一次投递
→ Spring Cloud Stream 函数式 Consumer
→ 本地事务 INSERT Inbox(event_id, consumer_name)
→ 唯一冲突：重复消息，跳过业务动作并确认
→ 首次消息：执行领域业务写入
→ 更新 Inbox.processed_at
→ 同一事务提交后由 Binder 确认
```

失败语义：

- 业务动作异常时，业务写入与 Inbox INSERT 一起回滚。
- RocketMQ 可以重新投递，下一次仍能成功插入 Inbox。
- 消费成功但确认失败造成重复投递时，Inbox 唯一约束跳过重复业务动作。
- Inbox 只提供事件级幂等；库存、工单、质量和设备命令仍必须使用状态机、条件更新和业务唯一约束。

## 7. Outbox DEAD 与 RocketMQ DLQ

两者是不同故障面：

| 状态 | 含义 | 责任方 |
|---|---|---|
| Outbox `DEAD` | Producer 多次无法把事件交给 Broker | 生产服务、数据库与 Broker 连接、消息大小或配置 |
| RocketMQ `%DLQ%group` | Broker 已持有消息，但消费者持续失败 | 消费服务、事件兼容、业务约束或依赖系统 |

必须分别提供指标、告警、诊断和人工重放流程。重放时必须复用原始 `eventId`，并保留操作审计。

## 8. Integration Hub

Integration Hub 负责：

- 外部端点和 Client 配置。
- 外部模型到 MOM 契约的映射。
- 请求与响应审计。
- 幂等键和重复请求处理。
- 重试、退避、死信、人工补偿和对账。
- 入站与出站限流。
- 外部错误码与内部错误模型转换。

Integration Hub 不负责：

- 修改其他领域数据库。
- 承担 MES/WMS/QMS 核心业务规则。
- 使用一个万能流程表编排所有业务。

## 9. Feign 调用规则

- 必须设置连接和读取超时。
- 只传递白名单上下文，例如用户、Client、工厂、Trace 和关联标识。
- 禁止透传原始外部 Header。
- 失败重试只用于幂等查询或明确可重试操作。
- 写请求默认不进行客户端无条件重试。
- 调用方必须区分业务拒绝、超时、服务不可用和未知错误。

## 10. Seata 边界

允许：

- 时间短、参与资源明确、失败可快速回滚的 PoC 场景。
- 少量关键资源预留的 TCC 评估。

禁止：

- 供应商送货到成品入库的全链路事务。
- 等待人工检验、设备执行或外部系统回调的事务。
- 用 Seata 替代 Outbox、幂等、对账和人工补偿。

## 11. 错误恢复

所有跨系统流程必须定义：

- 超时。
- 重试次数和退避策略。
- 可重试与不可重试错误。
- Outbox DEAD、RocketMQ DLQ 和人工处理入口。
- 对账方式。
- 最终业务状态。
- Trace、日志和告警。
