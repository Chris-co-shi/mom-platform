# 领域边界

## 1. 原则

- 每个领域服务拥有自己的权威数据和业务规则。
- 服务只能修改自身 Schema 中的数据。
- 跨领域同步查询使用 `*-client`；跨领域状态传播优先使用领域事件。
- Traceability 是跨域追溯查询模型的权威服务，但不反向修改 MES、WMS 或 QMS 事实。

## 2. 权威数据归属

| 领域 | 权威数据 | 不应拥有 |
|---|---|---|
| IAM | 用户、Client、角色、权限、数据范围、安全审计 | 物料、工单、库存 |
| MDM | 集团、工厂、组织、物料、供应商、客户、单位、版本索引 | 工单执行状态、库存余额 |
| MES | 生产需求、工单、版本快照、投料、过程记录、生产产出 | 仓库余额、检验结论 |
| WMS | 仓库、库区、库位、容器、库存流水、余额、预占和锁定 | 工艺、配方、检验规则 |
| QMS | 检验任务、样本、结果、放行、不合格处置、偏差和 CAPA | 库存数量、生产工单主体 |
| EMS | 能源点位、采集结果和统计 | 设备维修履历 |
| EAM | 设备台账、点检、保养、维修、备件和履历 | 生产批次谱系 |
| Integration | 外部端点、映射、请求记录、Outbox/Inbox、重试、补偿和对账 | 领域业务最终状态 |
| Traceability | 谱系边、追溯查询模型、影响分析和召回结果 | 原始库存和检验事实 |

## 3. 关键边界场景

### 3.1 工单下达

- MES 保存工单和生产版本快照。
- MDM 提供版本内容或版本索引。
- WMS 不保存工单主数据，只保存预占关联和业务单号。

### 3.2 来料放行

- QMS 保存检验结论和放行事实。
- WMS 消费放行事件并调整库存质量状态。
- QMS 不直接更新 WMS 库存表。

### 3.3 原料投料

- MES 保存投料业务事实。
- WMS 保存库存消耗流水。
- Traceability 保存输入批次与输出批次关系。
- 使用事件和幂等保证最终一致，不由 MES 跨库写 WMS。

### 3.4 WCS 自动入库

- WMS 拥有入库业务请求和最终库位关系。
- WCS 拥有运输任务、路由和设备执行状态。
- WCS 返回业务结果，不直接修改 WMS 数据库。

## 4. API / Client / Server 边界

### `*-api`

允许：

- Command、Query、DTO、事件契约、枚举和错误码。

禁止：

- Entity、Mapper、Repository、数据库注解和实现类。

### `*-client`

允许：

- Feign Client、调用适配、错误映射和客户端自动配置。

禁止：

- 业务状态机、数据库访问和服务端实现。

### `*-server`

允许：

- Interface、Application、Domain、Infrastructure 实现。

禁止：

- 直接依赖其他领域的 `*-server`。

## 5. 领域事件命名

领域事件使用已经发生的事实命名，例如：

- `IncomingInspectionPassed`
- `InventoryReceived`
- `ProductionOrderReleased`
- `MaterialReserved`
- `MaterialConsumed`
- `FinishedBatchProduced`
- `FinishedInspectionReleased`
- `PutawayCompleted`

禁止使用含糊的 `DataChanged`、`ProcessMessage` 等名称。

## 6. 边界变更规则

新增跨领域依赖前必须回答：

1. 数据权威属于哪个领域？
2. 是即时校验还是状态传播？
3. 是否可以使用本地副本或查询模型？
4. 失败时如何重试、补偿和对账？
5. 是否需要新增 ADR？
