## 1. 当前真实状态

P1.5 S00～S12 已完成并合并：Authorization Server、四个 Public Client、RBAC/Scope、Session 与 Refresh Rotation、Gateway/业务 Resource Server、`/api/iam/me`、管理 API、Web/Mobile Runtime 和安全 E2E 均已交付。Android Keystore、HTTPS App Link 与真机强杀恢复作为 Phase 02 Mobile 正式联调前置验收项。

## 2. 认证模型

所有用户客户端采用 Authorization Code + PKCE S256 + OpenID Connect：

| Client ID | 应用 | 用户类型 |
|---|---|---|
| `mom-admin-web` | MOM Admin | `INTERNAL` |
| `mom-supplier-web` | Supplier Portal | `SUPPLIER` |
| `mom-customer-web` | Customer Portal | `CUSTOMER` |
| `mom-mobile-pda` | MOM Mobile PDA | 具备 Mobile Access 的 `INTERNAL` |

全部是 Public Client，无 `client_secret`，Redirect URI 精确匹配，不开放动态 Client 注册。

不建设 BFF。IAM Cookie 只维护 IAM 页面状态和基础 SSO；Gateway API 统一使用 Bearer Access Token。

## 3. Token 与 Session

- Access Token：10 分钟短期 JWT。
- Refresh Token：高熵 Opaque Token，数据库只保存 HMAC-SHA-256 摘要。
- ID Token：不得用于调用业务 API。
- Web Session：8 小时绝对有效期。
- Mobile Session：12 小时绝对有效期。
- 一个 Session 最多一个 ACTIVE Refresh Token。
- 旧 Refresh Token 重放：Session → `COMPROMISED`，撤销当前 Token，`sid` 写入 Redis。
- Redis 不可用：受保护 API Fail Closed。

## 4. 授权模型

采用 `User → Role → Permission`：

- 不允许用户直接分配 Permission。
- 不支持角色继承、Deny Permission 或 ABAC。
- Permission Code 使用 `domain:resource:action`。
- Role 限定 `applicable_user_type`。
- Factory Scope 与 Role 分离。
- P1.5 通用数据权限只实现 Factory Scope 和 Party Scope。

`X-Factory-Id` 只是当前工作上下文，不是授权证明。外部用户的 `party_id` 来自 JWT，不允许前端自由切换。

## 5. 安全职责

### IAM

负责认证、Client、用户、角色、Permission、Factory Scope、Party Binding、Token、Session、撤销、安全审计、`/api/iam/me` 和管理 API。

### Gateway

负责 JWT 协议校验、Client 粗粒度入口、revoked sid、CORS、路由、限流、Correlation ID、Trace、删除伪造 `X-MOM-*` Header，并原样转发 Bearer JWT。

Gateway 不负责最终业务 Permission、Factory/Party 对象归属、业务状态机或动态菜单。

### 业务服务

作为 Resource Server 二次校验 Token，负责最终 Permission、Factory Scope、Party Scope、对象归属、领域状态、授权和业务审计。

Phase 02 对外服务通过 `phase02` Profile 显式启用 Resource Server；缺少 `IAM_ISSUER` 或 `IAM_JWK_SET_URI` 时启动失败。`/api/mdm/**`、`/api/wms/**`、`/api/qms/**`、`/api/integration/**` 都不能只依赖 Gateway 判断。`/internal/**` 也不得依赖“端口不可达”的假设：技术探针默认不注册，服务间正式身份与网络策略在业务接口落地时按场景冻结。

## 6. Web 与 Mobile

Web：Access Token 与 Refresh Token 都只存在内存；使用 Single Flight Refresh；`/api/iam/me` 初始化权限上下文；不使用 localStorage、IndexedDB、持久化 Pinia、普通 Cookie、hidden iframe 或跨标签 Token 同步。

Mobile：使用系统浏览器 + PKCE + HTTPS App Link；Access Token 在内存，Refresh Token 在 Android 安全存储；冷启动优先刷新恢复；离线命令绑定用户、Session、Client、Factory、Party 和 Permission。

## 7. CurrentActor 与数据审计

P1.5 S01 在 `mom-core`、`mom-security`、`mom-data` 间建立无反向依赖的 CurrentActor 和 MyBatis-Plus 自动填充边界。业务归属字段不得由 `MetaObjectHandler` 自动填充，后台任务必须显式建立 SYSTEM Actor。

## 8. 安全审计

安全审计与业务审计分离。不得记录密码、密码摘要、Access Token、Refresh Token、Refresh Token 摘要、Authorization Code、`code_verifier`、完整 Authorization Header 或 JWT 私钥。

## 9. 后续实施

P1.5 已封板；Phase 02 从三个仓库最新、干净的 `main` 创建，不从旧 P1.5 阶段分支继续演进。
