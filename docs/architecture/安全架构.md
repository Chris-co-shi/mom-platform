# 安全架构

## 1. 安全目标

- 统一认证和授权。
- 外部系统、用户和设备身份相互隔离。
- 功能权限与工业数据范围同时生效。
- 敏感操作可审计。
- 密钥、Token 和敏感数据不泄漏到代码、日志和 Trace。

## 2. 授权模式

| 客户端 | 授权方式 |
|---|---|
| MOM Web | Authorization Code + PKCE |
| 供应商门户 | Authorization Code + PKCE |
| 客户门户 | Authorization Code + PKCE |
| PDA | Authorization Code + PKCE，使用系统浏览器或安全 WebView |
| 内部服务 | Client Credentials |
| ERP Simulator | Client Credentials，可扩展签名和 IP 白名单 |
| PCS/WCS | 独立机器 Client + Client Credentials |

禁止使用 OAuth2 Resource Owner Password Credentials Grant。

## 3. Token 策略

- Access Token 使用短期 JWT。
- JWT 必须校验签名、Issuer、Audience、过期时间和 Scope。
- Refresh Token 由 Authorization Server 管理并支持轮换、吊销和重放检测。
- 服务只接受面向自身或平台 API 的 Audience。
- 高风险操作可要求更高 Scope 或二次确认扩展。

## 4. Gateway 安全职责

Gateway 负责：

- Token 基础校验。
- 路由级 Scope 和公开接口控制。
- Redis 分布式限流。
- 注入白名单内部上下文。
- 统一 401、403、429 和安全审计。

Gateway 不负责：

- 最终领域对象级授权。
- 数据库数据范围 SQL 规则。
- MES/WMS/QMS 业务条件判断。

## 5. 服务端授权

每个领域服务必须再次执行：

- Token 与 Audience 校验。
- 方法或资源权限校验。
- 工厂、车间、产线、仓库、库区等数据范围校验。
- 领域状态和业务规则校验。

不能只信任 Gateway 注入的 Header；内部上下文必须具有可信来源或签名保护。

## 6. 数据权限

统一数据范围模型至少支持：

```text
集团
工厂
车间
产线
仓库
库区
质量实验室
设备区域
本人/所属组织
```

数据权限实现要求：

- 规则接口与 SQL 重写机制解耦。
- 查询和写操作均需校验，不只对列表查询生效。
- 不允许通过传入其他工厂 ID 绕过范围。
- 数据权限规则应有自动化测试。

## 7. Redis 限流

限流维度至少包括：

- route。
- IP。
- user。
- client。
- factory。
- supplier。
- customer。
- device。

接口分级：

- 登录、验证码等公开安全接口：优先保护系统。
- 外部 API：Redis 故障时倾向保守拒绝或降级。
- 生产操作：不得因 Redis 短时故障简单全部失败，应提供受控本地应急策略。
- 报表和追溯：使用成本型令牌。

## 8. 密钥和配置

- 密钥不得提交到 Git。
- 本地使用 `.env` 或开发密钥文件，仓库只提供示例。
- k3s 使用 Secret 或后续外部密钥管理组件。
- 日志中禁止输出 Authorization、Cookie、Client Secret 和数据库密码。
- 密钥轮换必须不要求重新构建镜像。

## 9. 审计

以下行为必须审计：

- 登录成功、失败、登出和 Token 吊销。
- 用户、角色、权限、Client 和数据范围修改。
- 质量放行、让步接收、报废和返工。
- 库存调整、解锁和手工冲销。
- 工单取消、人工接管和设备命令干预。
- 重试、补偿、对账确认和召回操作。

审计至少记录：操作者、Client、来源 IP、工厂、动作、对象、结果、时间、业务单号和关联标识。

## 10. 安全测试

- 未认证和过期 Token。
- Audience、Scope 和角色不匹配。
- 跨工厂和跨仓库越权。
- 重放请求和重复幂等键。
- 限流绕过。
- Header 伪造。
- 敏感字段日志扫描。
- 依赖漏洞和 SBOM 检查。
