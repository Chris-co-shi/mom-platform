# V1 路线图

## 1. 总目标

交付一个可部署、可重复演示的工业 MOM V1，完成供应商送货到客户发运及批次追溯的最小完整闭环，并在三节点 k3s 环境完成可观测性、滚动升级、回滚和故障恢复演练。

## 2. 阶段状态

| 阶段 | 状态 | 目标 | 主要输出 |
|---|---|---|---|
| Phase 01 | **Completed — Foundation Only（2026-07-19）** | 基础技术骨架可运行 | JDK 25、Boot 4、Gateway/IAM/MDM/Integration 骨架、数据库、中间件、Trace、日志、指标、告警 |
| P1.5 | **Completed / Merged（S00～S12）** | 认证与授权闭环 | IAM、OAuth/OIDC、PKCE、RBAC、Scope、Session、Gateway/Resource Server、Web/Mobile Auth、安全 E2E |
| Phase 02 | **Pending / Ready after preflight cleanup** | 到货、检验和入库闭环 | 供应商送货、来料检验、PDA 收货、库存流水和余额 |
| Phase 03 | Planned | 生产执行和 PCS 协同闭环 | 工单、版本快照、预占、投料、半成品/成品、PCS 模拟器 |
| Phase 04 | Planned | 放行、WCS 入库、发运和追溯闭环 | 成品放行、自动入库、客户发运、正反向追溯、模拟召回 |

## 3. Phase 01：基础技术骨架

### 状态

已完成基础技术骨架。详见：[Phase 01 完成报告](Phase-01-完成报告.md)。

### 已输出

- Maven Reactor 和统一依赖管理。
- Gateway、IAM、MDM、Integration 最小启动应用。
- Spring Security / OAuth2 Resource Server 依赖基线。
- PostgreSQL 每服务独立 Schema 和 Flyway 规则。
- Nacos 注册发现。
- RocketMQ 生产、消费、重试、DLQ 与 Trace 上下文传播。
- Redis 限流和幂等基础能力。
- Seata AT 受控短事务 PoC。
- OpenTelemetry Collector、Tempo、Prometheus、Loki、Alloy 和 Grafana。
- Testcontainers 集成测试和 ArchUnit 依赖约束。

### 边界

Phase 01 未完成正式 Authorization Server、完整 Gateway/业务服务 Resource Server、用户/角色/权限/Session、Refresh Rotation、Web/Mobile 登录或安全 E2E。这些内容统一进入 P1.5。

## 4. P1.5：认证与授权闭环

### 状态

S00～S12 已完成并合并。详见：

- [P1.5 认证与授权设计基线](../security/P1.5-认证与授权设计基线.md)
- [P1.5 认证与授权闭环实施计划](P1.5-认证与授权闭环计划.md)

### 输出

- `INTERNAL`、`SUPPLIER`、`CUSTOMER` 用户与 Party Binding。
- 四个 Public Client 与 Authorization Code + PKCE + OIDC。
- Access Token、Opaque Refresh Token、ID Token、JWT Claims。
- RBAC、Permission、Factory Scope、Party Scope。
- 用户授权 Session、Refresh Rotation、重放检测和撤销。
- Gateway 和业务服务双层 Resource Server。
- `/api/iam/me`、IAM 管理 API、安全审计和 Session 管理。
- CurrentActor 与 MyBatis-Plus 数据审计基础。
- MOM Admin、Supplier Portal、Customer Portal 与 Mobile Auth Runtime。
- 安全 E2E 与跨仓库一致性验收。

### 门禁

P1.5 安全门禁已经完成。Phase 02 业务服务启用 `phase02` Profile 后必须自身验证 JWT Issuer、Audience、Client/user_type、Permission、Factory/Party Scope、对象归属与 404 防枚举；Gateway 只做粗粒度入口控制。

## 5. Phase 02：供应商到货、质量与入库

### 状态

Pending / Ready after preflight cleanup。Android Keystore、HTTPS App Link 与真机强杀恢复是 Mobile 正式联调前置验收项，不再阻塞 P1.5 完成事实。

### 输出

- 供应商、物料、工厂、仓库、库位和容器最小主数据。
- 送货单、收货任务和来料检验任务。
- PDA 扫码收货与离线队列最小实现。
- 检验合格、拒收、降级和让步接收。
- 库存事实流水、实时余额、质量状态和幂等提交。

### 推荐首个垂直切片

```text
供应商送货通知
  → WMS 创建收货任务
  → PDA 或受控接口完成实物收货
  → WMS 创建收货批次、容器和 INSPECTION_HOLD 库存事实
  → 发布 MaterialReceived 事实事件
  → QMS 创建并完成来料检验任务与处置
  → 发布 InspectionDispositionDecided 事实事件
  → WMS 调整为 AVAILABLE、REJECTED、CONCESSION 或 DOWNGRADED
```

首个切片必须同时具备状态机、幂等请求、本地事务 + Outbox、Consumer Inbox、Trace/日志/指标、权限与 Factory/Party 授权，以及重复提交、异常回滚、消息重复和越权验收。

### 完成定义

同一收货请求重复提交三次，只产生一次收货事实、一次库存流水和一次余额增加；未经质量放行的批次不可进入可用库存；跨用户类型、供应商、工厂或权限的请求被正确拒绝。

## 6. Phase 03：生产执行与 PCS

### 输出

- 配方、BOM、工艺和质量标准版本索引。
- 生产工单和不可变生产版本快照。
- 原料预占、领用、投料和消耗。
- PCS 命令协议、状态机和软件模拟器。
- 半成品、成品批次和容器产出。
- 投料和产出谱系边。

### 完成定义

PCS 断线、超时和重复回执后，工单能够恢复并继续执行，且库存、投料和产出不重复。

## 7. Phase 04：成品放行、自动仓储和追溯

### 输出

- 成品检验与自动/人工放行。
- WCS 自动入库请求、运输任务和模拟设备。
- 成品库存、客户订单、拣货和发运。
- 正向追溯、反向追溯、影响分析和模拟召回。
- k3s 部署、滚动升级、回滚和故障演练报告。

### 完成定义

可以由任一成品批次查到所有原料来源，也可以由任一原料批次查到受影响成品、当前库存、发运单和客户。

## 8. 统一里程碑门禁

每个阶段进入下一阶段前必须通过：

1. 需求验收。
2. 自动化测试。
3. 架构与安全边界检查。
4. 数据迁移和中文注释检查。
5. Trace、日志和指标检查。
6. 故障与安全场景检查。
7. 文档更新与跨仓库一致性检查。

## 9. 延期处理原则

若阶段延期，优先减少非核心页面和模拟数据丰富度，不得删除库存一致性、批次谱系、Outbox/Inbox、幂等、PCS/WCS 状态机、认证授权边界、链路追踪、Redis 限流或故障恢复验收。
