# 文档维护约定

本文档定义 `mom-platform` 仓库的文档工作分支、命名方式、维护流程和 README 风格要求。

## 1. 固定文档分支

所有文档相关操作统一在以下分支进行：

```text
agent/complete-chinese-docs
```

适用范围包括：

- 新增文档。
- 修改文档。
- 重命名文档。
- 调整目录与导航。
- 更新 README。
- 更新 ADR。
- 更新需求、计划和架构说明。
- 创建或更新文档 PR。

禁止直接在 `main` 分支修改文档。

## 2. 标准工作流程

```text
同步 main 最新内容
→ 在 agent/complete-chinese-docs 修改文档
→ 检查链接、命名和内容一致性
→ 创建 Draft PR
→ 完成审查
→ Squash merge 到 main
→ 继续在固定文档分支开展下一次文档工作
```

## 3. 命名原则

- 文档文件名优先使用中文，便于阅读和检索。
- `requirements`、`plans`、`architecture`、`adr` 等一级目录保留英文，以兼顾脚本和跨平台工具。
- ADR 保留编号前缀，例如：`ADR-006-库存流水余额与预占模型.md`。
- 计划文件保留阶段编号，例如：`Phase-01-技术骨架计划.md`。
- 避免含义不清的名称，例如 `说明.md`、`其他.md`、`临时.md`。

## 4. README 风格要求

根目录 `README.md` 是开源项目首页，应保持现代化、清晰且可信，至少包含：

1. 项目名称、定位与一句话价值描述。
2. CI、Java、Spring Boot、项目状态等徽章。
3. 文档、需求、计划、架构和 ADR 快速导航。
4. 当前成熟度和非生产可用提示。
5. V1 端到端业务闭环。
6. Mermaid 系统全景图。
7. 核心能力与权威模块矩阵。
8. 技术基线。
9. 仓库和模块结构。
10. 快速开始。
11. 文档导航。
12. 当前路线图。
13. MOM 项目仓库族。
14. 架构原则和开源复用原则。

README 不得：

- 宣称尚未实现的功能已经完成。
- 堆积大量无关徽章或营销文案。
- 复制详细需求和架构正文。
- 使用已经失效的链接。
- 将根 README 变成开发日志。

## 5. 文档一致性

- 根 README 负责项目入口，不替代详细文档。
- `docs/README.md` 是文档导航权威入口。
- 需求、计划、架构和 ADR 各自承担不同职责，禁止重复维护同一事实。
- 代码与文档冲突时，必须明确是实现偏离还是决策变化。
- 架构决策发生变化时，新建 ADR 替代旧 ADR，不静默覆盖历史结论。

## 6. PR 要求

文档 PR 至少说明：

- 修改了什么。
- 为什么修改。
- 影响哪些需求、计划、架构或 ADR。
- 是否只涉及文档。
- 如何验证 Markdown 链接、Mermaid 和内容一致性。

多个零散文档提交合并时，优先使用 `Squash merge` 保持主分支历史清晰。
