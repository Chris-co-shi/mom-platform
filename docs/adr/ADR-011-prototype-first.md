# ADR-011：Web 与 PDA 原型先行

- 状态：Accepted
- 日期：2026-07-18
- 关联需求：所有包含 Web/PDA 交互的业务需求
- 关联计划：[V1 垂直切片计划](../plans/V1垂直切片计划.md)

## 1. 背景

Vben Admin 和 uni-app 只提供实现基座，不能替代业务流程和交互设计。工业软件页面包含复杂状态、扫码、设备异常、离线、人工接管和数据权限，如果直接从数据库表和接口开始开发，容易形成可用性差的 CRUD 页面。

## 2. 候选方案

- 后端完成后前端直接按接口开发：启动快，但页面流程、状态和接口容易反复修改。
- 只输出文字页面清单：能说明范围，但无法验证布局和交互。
- 每个垂直切片先完成流程、状态和可视化原型：前期增加设计工作，但减少开发返工。

## 3. 决策

每个前端垂直切片必须按以下顺序推进：

```text
业务目标
→ 用户角色与操作流程
→ 页面清单
→ 页面状态矩阵
→ Web/PDA 原型图
→ 组件映射
→ API 与权限映射
→ 开发
→ UI 验收
```

原型分别存放在 `mom-web` 和 `mom-mobile` 仓库；`mom-platform` 的需求和计划文档引用原型版本。

## 4. 强制产物

- 用户流程图。
- 页面清单。
- 正常、加载、空数据、无权限和错误状态。
- PDA 在线、离线、同步中、冲突和完成状态。
- 关键弹窗和人工接管状态。
- Vben/Ant Design Vue、uni-app 和自研组件映射。
- 查询 API、命令 API、权限、数据范围、幂等键和错误码映射。

## 5. 后果

正向：前后端契约更稳定，工业异常状态被提前发现，适合 AI 编码按明确原型实现。

负向：开发前需要额外设计时间，原型和实现必须同步维护。

## 6. 验证方式

- PR 模板检查是否关联原型和页面状态矩阵。
- 前端功能验收同时检查正常、异常和无权限状态。
- 接口变更时检查原型和 API 映射是否同步更新。

## 7. 替代条件

纯后端能力、无用户界面的基础设施切片可以不输出页面原型；一旦涉及 Web、门户或 PDA 交互，本 ADR 强制生效。
