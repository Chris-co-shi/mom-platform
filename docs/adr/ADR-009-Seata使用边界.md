# ADR-009：Seata 使用边界

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[集成架构](../architecture/集成架构.md)

## 1. 背景

项目技术目标包含 Seata，但制造流程包含人工检验、设备执行、外部回调和长时间等待。如果把完整流程放入分布式事务，会造成锁时间过长、恢复困难和服务强耦合。

## 2. 候选方案

- 全面使用 Seata AT：编码直观，但不适合长流程和设备交互。
- 完全不使用 Seata：边界简单，但无法展示短跨库事务和 TCC 能力。
- 限定使用 AT/TCC，长流程使用事件和补偿：兼顾技术验证和工程合理性。

## 3. 决策

- 本地事务是默认选择。
- Seata AT 只用于时间短、参与资源明确且可快速回滚的同步场景。
- TCC 只在库存预留等具有明确 Try/Confirm/Cancel 语义的关键资源场景评估。
- 完整制造流程使用 Outbox、RocketMQ、Inbox、幂等、状态机、对账和人工补偿。
- 不允许等待人工、设备或外部系统时保持 Seata 全局事务。

## 4. 使用前检查

引入 Seata 前必须回答：

1. 事务持续时间是否可控？
2. 参与服务和数据库是否明确？
3. 是否存在人工或设备等待？
4. 回滚是否真正符合业务含义？
5. 是否可以使用本地事务加事件替代？

## 5. 后果

正向：保留 Seata 的学习和展示价值，同时避免把长业务流程错误建模为数据库事务。

负向：开发者需要同时理解本地事务、最终一致、TCC 和补偿边界。

## 6. 验证方式

- Phase 01 完成一个受控 AT PoC。
- 对库存预占评估 TCC，但只有在必要时进入业务实现。
- 代码评审检查全局事务持续时间和调用链。
- 故障演练验证事件流程可重试、补偿和对账。

## 7. 替代条件

若 Boot 4 兼容性、维护成本或运行风险不可接受，可移除 Seata 运行时，但 Outbox/Inbox 和领域补偿设计不受影响。
