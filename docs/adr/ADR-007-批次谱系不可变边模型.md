# ADR-007：批次谱系不可变边模型

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[数据架构](../architecture/数据架构.md)

## 1. 背景

电解液生产存在原料合并、半成品转移、灌装拆分、返工、降级和让步接收。简单在成品表保存一个原料批次 ID 无法表达多对多关系，也无法支持召回影响分析。

## 2. 候选方案

- 在批次记录中保存父批次字段：只适合单父子关系。
- 使用可修改的树结构：难表达合并，历史纠错可能破坏审计。
- 使用不可变有向谱系边：能够表达多对多、拆分、合并和返工。

## 3. 决策

- 以批次为节点，以生产消耗/产出关系为有向边。
- 谱系边记录输入批次、输出批次、工单、工序、数量、单位、容器、时间和 `event_id`。
- 谱系边创建后原则上不可覆盖修改。
- 错误通过冲销边、更正边或新的业务事实表达。
- Traceability 通过领域事件维护跨域查询模型。

## 4. 支持场景

- 多批原料合并为一个中间批次。
- 一个中间批次拆分为多个成品批次。
- 返工批次重新投入。
- 不合格批次降级为其他用途。
- 由原料查成品、库存、发运和客户。
- 由成品查全部原料和生产路径。

## 5. 后果

正向：模型能自然表达复杂制造谱系，历史可审计，适合正反向查询。

负向：查询需要图遍历、闭环检测、数量守恒校验和性能优化。

## 6. 验证方式

- 构造至少一次合并和一次拆分。
- 验证返工不会形成非法循环；若业务允许回流，应使用新的批次节点。
- 在 10 万条谱系边数据下验证查询性能。
- 同一 `event_id` 重复处理不创建重复边。
- 模拟召回结果与库存和发运记录一致。

## 7. 替代条件

只有当关系规模或查询性能超过 PostgreSQL 可接受范围时，再评估图数据库；领域契约和不可变边原则保持不变。
