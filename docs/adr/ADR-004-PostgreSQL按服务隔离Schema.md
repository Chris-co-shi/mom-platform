# ADR-004：PostgreSQL 按服务隔离 Schema

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[数据架构](../architecture/数据架构.md)

## 1. 背景

V1 需要控制基础设施成本，同时保持领域数据归属和未来独立扩展能力。

## 2. 候选方案

- 所有服务共享表：开发快，但边界失效、跨域写入和耦合严重。
- 每服务独立数据库实例：隔离最好，但本地和 k3s 资源成本高。
- 共享 PostgreSQL 集群、每服务独立 Schema：兼顾资源和逻辑隔离。

## 3. 决策

- V1 使用一个 PostgreSQL 集群。
- IAM、MDM、MES、WMS、QMS、EMS、EAM、Integration、Traceability 使用独立 Schema。
- 每个服务只读写自身 Schema。
- 禁止跨 Schema JOIN、外键和写入。
- 数据迁移由各服务自己的 Flyway 管理。

## 4. 跨域查询

跨域查询采用 Query API、事件投影或明确的查询服务，不通过数据库 JOIN 绕过服务边界。

## 5. 后果

正向：资源占用低、备份运维集中，同时保留领域隔离。

负向：物理数据库故障仍影响多个服务；复杂跨域查询需要额外查询模型。

## 6. 验证方式

- 为服务账号配置最小 Schema 权限。
- 集成测试验证服务无法写入其他 Schema。
- 代码和 SQL 扫描禁止跨 Schema 引用。
- Flyway 版本在服务内独立推进。

## 7. 替代条件

当某个领域在容量、合规或可用性上需要独立数据库时，可以单独迁移；服务契约和领域边界不应因此变化。
