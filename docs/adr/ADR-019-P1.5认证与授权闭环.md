# ADR-019：P1.5 认证与授权闭环

- 状态：Accepted
- 日期：2026-07-19
- 替代：[ADR-003：OAuth2.1 与 OIDC 授权模型](ADR-003-OAuth2与OIDC授权模型.md)
- 关联文档：[P1.5 认证与授权设计基线](../security/P1.5-认证与授权设计基线.md)

## 1. 背景

Phase 01 已完成可运行的基础技术骨架，但旧文档把最小安全依赖、应用占位和部分上下文能力描述成“Gateway 与安全闭环”“IAM 最小 OAuth2.1/OIDC 服务已完成”。当前代码尚未形成正式 Authorization Server、完整 Gateway Resource Server、用户/角色/权限/Session 数据模型、Refresh Rotation、Web/Mobile 登录和安全 E2E。

同时，MOM 需要统一内部用户、供应商用户、客户用户、MOM Admin、供应商门户、客户门户和 Android PDA 的认证授权协议，避免后端、Web 与 Mobile 各自演进出互相冲突的方案。

## 2. 候选方案

### 2.1 BFF + 应用 Session Cookie

优点：浏览器不接触 Token。

缺点：引入 BFF、应用 Session、CSRF、跨应用会话和额外部署边界，不符合本阶段冻结结论。

### 2.2 SPA/Native Public Client + Authorization Code + PKCE

优点：使用标准 OAuth/OIDC 协议，四个应用 Client 边界清晰，Gateway API 统一 Bearer Token，Web 与 Mobile 可以共享后端协议。

缺点：需要严格设计 Token 存储、Refresh Rotation、Session 撤销和浏览器/移动端恢复流程。

### 2.3 自研账号密码 + JWT

优点：短期实现简单。

缺点：协议、安全、Session、撤销、Client 隔离和长期维护风险不可接受。

## 3. 决策

采用方案 2.2，并冻结以下结论：

1. 统一 `iam_user`，`user_type` 仅为 `INTERNAL`、`SUPPLIER`、`CUSTOMER`。
2. 预置 `mom-admin-web`、`mom-supplier-web`、`mom-customer-web`、`mom-mobile-pda` 四个 Public Client。
3. 全部用户客户端采用 Authorization Code + PKCE S256 + OpenID Connect，无 `client_secret`，Redirect URI 精确匹配。
4. 不建设 BFF；Gateway API 只接受 Bearer Access Token。
5. Access Token 使用 10 分钟短期 JWT；Refresh Token 使用高熵 Opaque Token；ID Token 不调用业务 API。
6. Web 用户授权 Session 绝对有效期 8 小时，Mobile 为 12 小时，Rotation 不延长绝对期限。
7. Refresh Token 数据库存 HMAC-SHA-256 摘要；一个 Session 最多一个 ACTIVE Token；旧 Token 重放使 Session `COMPROMISED`。
8. JWT `sid` 对应 `iam_user_session`；Redis 保存 revoked sid；Redis 不可用时受保护 API Fail Closed。
9. 授权模型采用 User → Role → Permission；只实现 Factory Scope 和 Party Scope。
10. Gateway 只做协议与粗粒度入口安全；业务服务负责最终 Permission、Factory/Party Scope、对象归属和领域授权。
11. Web Token 只存在内存；Mobile Access Token 在内存、Refresh Token 在 Android 安全存储。
12. `/api/iam/me` 是 Web/Mobile 正式权限上下文来源。
13. CurrentActor 与 MyBatis-Plus 审计基础在 P1.5 S01 实现。
14. P1.5 正式 Flyway DDL 必须包含完整准确的中文注释。

完整细节以 [P1.5 认证与授权设计基线](../security/P1.5-认证与授权设计基线.md) 为唯一权威来源。

## 4. 职责边界

- `mom-platform`：后端安全协议、IAM、Gateway、Resource Server、RBAC、Scope、Session、审计和管理 API。
- `mom-web`：浏览器 PKCE、内存 Token、Single Flight Refresh、Access/UI 体验控制和三个 Web Client。
- `mom-mobile`：系统浏览器 PKCE、App Link、安全存储、冷启动恢复和离线命令身份归属。

前端或移动端权限判断都不是最终安全边界。

## 5. 后果

### 正向

- 三仓库共享一个可测试的安全协议。
- 用户类型、Client、Role、Permission、Factory 与 Party 边界清晰。
- Access Token 可短期无状态校验，同时通过 `sid` 支持立即撤销。
- Web 不持久化 Token，Mobile 使用平台安全存储。
- 后续业务 Slice 不需要重复发明登录、权限和数据范围方案。

### 负向

- P1.5 必须在业务闭环前投入 12 个 Slice。
- Refresh Rotation、重放检测、撤销和跨服务授权测试复杂度较高。
- Redis revoked sid 检查成为受保护 API 的关键安全依赖。
- Web 页面刷新需要重新执行顶层 PKCE，而不是从持久化 Token 恢复。

## 6. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 前端把隐藏按钮当安全边界 | 业务服务强制最终授权并提供越权测试 |
| `X-Factory-Id` 被当作授权证明 | 服务端始终与 JWT `factory_ids` 和对象归属交叉校验 |
| Refresh 并发或网络不确定导致重放 | 单 ACTIVE Token、事务行锁、Single Flight、旧 Token 不重试 |
| Redis 故障放过已撤销 Session | 受保护 API Fail Closed |
| 文档与实现再次漂移 | S12 执行三仓库 E2E 与契约一致性检查 |

## 7. 验证方式

- Client 与 user_type 入口隔离测试。
- PKCE、nonce、state、Redirect URI 精确匹配测试。
- JWT 签名、Issuer、Audience、时间和 Claims 测试。
- Refresh Rotation、并发刷新、旧 Token 重放与 Session 撤销测试。
- Redis 故障 Fail Closed 测试。
- Permission、Factory、Party、对象归属和 404 防枚举测试。
- Web 刷新恢复、Mobile 冷启动和离线命令跨用户隔离测试。
- 日志与安全审计敏感信息扫描。

## 8. 替代条件

只有当标准 OAuth/OIDC 实现无法满足 MOM 的关键协议、安全或维护要求时，才允许通过新 ADR 评估外部 IAM 或其他架构。不得回退到 Resource Owner Password Credentials、自研密码 JWT、浏览器持久化 Token 或把 Gateway Header 作为唯一授权依据。
