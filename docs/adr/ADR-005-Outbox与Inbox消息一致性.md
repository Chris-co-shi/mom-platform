# ADR-005：Outbox 与 Inbox 消息一致性

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[集成架构](../architecture/集成架构.md)

## 1. 背景

生产、库存、质量、Integration、PCS 和 WCS 之间存在大量跨服务长流程。使用数据库事务和 MQ 双写会产生业务已提交但消息丢失，或消息已发送但业务回滚的问题。

## 2. 候选方案

- 直接数据库写入后发送 MQ：简单，但存在双写窗口。
- 所有流程使用 Seata：无法覆盖人工、设备和长时间流程，且耦合过高。
- Transactional Outbox + Inbox：接受至少一次投递，通过本地事务和幂等实现最终一致。

## 3. 决策

- 需要可靠发布领域事件的服务使用 Outbox。
- 消费者使用 Inbox 或等价唯一约束去重。
- 业务写入与 Outbox 写入处于同一本地事务。
- 消息携带唯一 `event_id`、事件类型、版本和关联标识。
- 消费端业务写入与 Inbox 记录处于同一本地事务。

## 4. 失败处理

必须覆盖：

- 业务提交后应用崩溃。
- MQ 发送成功但 Outbox 状态更新失败。
- 消息重复投递。
- 消费处理成功但确认失败。
- 长时间未发布、重试耗尽和死信。

## 5. 后果

正向：不依赖跨服务长事务，可恢复、可审计、适合设备和外部系统流程。

负向：引入最终一致延迟、Outbox 清理、重试和监控成本。

## 6. 验证方式

- 同一消息重复投递三次，只产生一次业务结果。
- 业务提交后停止发布器，恢复后消息仍能发送。
- 模拟发送成功但状态更新失败，重复发布仍不重复处理。
- Grafana 能查看 Outbox 堆积、重试和消费延迟。

## 7. 替代条件

若未来采用具备成熟事务日志 CDC 的平台，可通过新 ADR 替代 Outbox 发布机制；消费幂等和业务事件契约仍然保留。
