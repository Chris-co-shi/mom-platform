# ADR-010：开源复用与许可证规则

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[开源来源登记](../open-source/source-origin.md)

## 1. 背景

项目会研究 Pig、Yudao Cloud、JetLinks、OpenWMS、Vben、uni-app、PLC4X、Milo 等开源项目。直接复制整套框架会把无关领域、旧技术栈和许可证义务带入 MOM，也不符合重新设计知识产权的目标。

## 2. 复用分类

### 直接依赖

适用于标准协议、成熟 SDK 和 UI 基座，例如 Spring、OpenTelemetry、Milo、PLC4X、Vben、uni-app。必须通过正式依赖或上游仓库引入。

### 学习后重构

适用于 Pig 的 Gateway/SAS/Feign 机制、Yudao 的数据权限机制、JetLinks 的协议 SPI 和设备会话。

### 仅领域研究

适用于 OpenWMS、EAM 项目的领域概念和流程参考。

### 必须自研

MOM 核心领域、批次谱系、库存一致性、质量规则、Integration Hub、PCS/WCS 状态机和故障恢复。

## 3. 决策

- 禁止通过批量改包名复制其他开源项目。
- 引入依赖前必须记录项目、版本或 Commit、许可证、用途和 NOTICE 要求。
- 复制源码时必须保留原始版权头并审查许可证兼容性。
- 优先“阅读机制—建立 MOM 约束—定义自有接口—重新实现—增加自有测试”。
- 发布版本必须生成 SBOM 和第三方依赖清单。

## 4. 登记字段

至少记录：

- 项目和官方地址。
- 精确版本、Tag 或 Commit。
- License。
- 使用方式。
- 引入模块。
- 是否复制或修改源码。
- NOTICE 和源代码公开要求。
- 核验日期。

## 5. 后果

正向：降低许可证和架构污染风险，确保 MOM 领域模型独立。

负向：学习后重写需要更多时间，无法直接享受通用框架全部功能。

## 6. 验证方式

- PR 检查新增依赖是否更新来源登记。
- CI 生成 SBOM 并扫描许可证。
- 定期检查第三方版本和许可证变化。
- 对复制源码执行版权头和来源审查。

## 7. 替代条件

如果未来选择整体基于某个开源平台二次开发，必须新建 ADR，重新评估许可证、升级路径、领域污染和迁移成本。
