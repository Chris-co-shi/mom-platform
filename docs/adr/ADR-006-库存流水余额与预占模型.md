# ADR-006：库存流水、余额与预占模型

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[数据架构](../architecture/数据架构.md)

## 1. 背景

制造库存需要同时支持高效查询、批次追溯、并发预占、质量锁定和事后对账。只保存当前余额无法解释库存变化，只保存流水又难以满足实时查询性能。

## 2. 候选方案

- 仅保存余额：查询快，但难审计、难重算、难定位并发问题。
- 每次查询实时汇总流水：事实完整，但性能和并发成本过高。
- 不可变事实流水 + 实时余额 + 独立预占/锁定：同时满足审计、性能和并发控制。

## 3. 决策

- 每次库存变化写入事实流水。
- 同一事务内更新实时余额。
- 余额区分现有、可用、预占、锁定和隔离数量。
- 预占必须关联工单、发运需求等业务对象。
- 余额更新使用条件更新、版本号或明确行锁。
- Redis 锁可作为流量协调手段，但不能成为库存正确性的唯一保证。

基础关系：

```text
available = on_hand - reserved - locked - quarantine
```

## 4. 幂等

收货、预占、释放、领用、退料、入库和发运命令必须携带业务幂等键。相同命令重复执行不得重复写入有效流水或修改余额。

## 5. 后果

正向：库存可解释、可追踪、可重算，同时保持实时查询性能。

负向：需要维护流水和余额一致性、对账任务和并发测试。

## 6. 验证方式

- 从流水重算余额并比较差异。
- 同一命令重复三次只生成一次有效事实。
- 并发预占不允许可用量变为负数。
- 不合格、过期、锁定和隔离批次不可预占。
- 模拟事务失败后流水与余额同时回滚。

## 7. 替代条件

未来可以增加事件溯源或专用库存计算组件，但必须保留可审计事实、实时查询和幂等约束。
