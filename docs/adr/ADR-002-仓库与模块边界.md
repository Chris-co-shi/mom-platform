# ADR-002：仓库与模块边界

- 状态：Accepted
- 日期：2026-07-18
- 关联文档：[领域边界](../architecture/领域边界.md)

## 1. 背景

项目同时包含 MOM 业务平台、PCS、WCS、Web、PDA、ERP 模拟器和基础设施。若全部放入一个仓库，会增加构建、发布和权限耦合；若拆分过细，则一个月 V1 会被仓库管理成本拖慢。

## 2. 候选方案

- 单一 Monorepo：统一管理简单，但 Java、前端、设备协同和基础设施生命周期混杂。
- 每个微服务一个仓库：边界最细，但版本协调和本地联调成本过高。
- 按产品运行时边界拆为七个仓库：兼顾独立发布和 V1 协作效率。

## 3. 决策

建立七个仓库：

- `mom-platform`
- `pcs-platform`
- `wcs-platform`
- `mom-web`
- `mom-mobile`
- `erp-simulator`
- `mom-infra`

`mom-platform` 内按领域平台组织，每个核心领域采用 `api / client / server` 分层。

## 4. 实施约束

- `*-api` 只存放跨模块契约。
- `*-client` 只存放调用适配。
- `*-server` 实现领域、应用和基础设施。
- 领域 `server` 禁止直接依赖其他领域 `server`。
- PCS/WCS 协议实现不得进入 MOM Reactor。
- Framework 不得包含 MES/WMS/QMS 业务规则。

## 5. 后果

正向：仓库边界与部署边界一致，PCS/WCS 和前端可以独立演进。

负向：需要维护跨仓库契约版本和联调环境。

## 6. 验证方式

- 使用 Maven Reactor 验证 MOM 内模块构建。
- 使用 ArchUnit 检查禁止依赖。
- PR 检查是否将 PCS/WCS 或前端实现误放入本仓库。

## 7. 替代条件

当跨仓库契约协调成本显著高于独立发布收益时，再评估 Platform Monorepo；不得在 V1 中随意调整仓库边界。
